The Waiting Problem

A for loop is like a to-do list with a fixed number of items. You know exactly how many times you will repeat. But some situations do not have a known count.

  • Waiting for water to boil: you check every minute, but you do not know how many minutes it will take
  • Asking a user for a password: they might get it right on the first try, or the fifth
  • Searching for a parking spot: you drive around until one opens up

In all of these, you repeat until a condition changes, not for a fixed number of times. This is the problem a while loop solves.

How a while Loop Works

A while loop checks a condition before each repetition. If the condition is True, the body runs. Then it checks again. When the condition becomes False, the loop stops.

Common misconception: the condition is not checked continuously while the body runs. Think of a gatekeeper who checks your ticket at the door. Once you are inside, the gatekeeper does not follow you around. They only check again when you come back to the entrance.

For Loops vs While Loops

Both repeat code, but they answer different questions.

for loop

  • "Do this for each item"
  • You know the count or have a sequence
  • Automatically moves to the next item
  • Cannot accidentally run forever

while loop

  • "Keep going while this is true"
  • You do not know how many times
  • You must update the condition yourself
  • Can run forever if condition never changes

Tricky phrasing: students often think "stop when X" and write while X. But the condition describes when to keep going, not when to stop. "Stop when count reaches 5" becomes while count < 5.

The Infinite Loop Trap

Imagine a thermostat set to turn off the heater when the room reaches 72 degrees. But the thermometer is broken and always reads 65. The heater never turns off. That is an infinite loop.

  • The condition never becomes False because nothing inside the loop changes it
  • The program freezes, printing forever or doing nothing visible
  • Fix: always make sure something inside the loop moves toward making the condition False
count = 1
while count <= 5:
    print(count)
    count = count + 1  # This line prevents an infinite loop

break: The Emergency Exit

Sometimes a loop needs to stop before the condition naturally becomes False. break is the emergency exit: it ends the loop immediately, no matter what the condition says.

  • Searching a list: once you find what you need, there is no reason to keep looking
  • User types "quit": stop the loop right away without finishing the current iteration
  • After a break, Python jumps to the first line after the loop

Think of pulling the fire alarm in a building. Everyone exits immediately. You do not wait for the end of the workday.

8
Section 3 · Lecture

continue: Skip This Lap

Where break ends the whole loop, continue skips just the rest of the current iteration and jumps back to the condition check.

Think of running laps on a track. You trip and decide to skip the rest of this lap, but you do not leave the track. You walk back to the start line and run the next lap normally.

Input Validation: The Gatekeeper

Think of a bouncer at a venue checking IDs. If your ID is not valid, you go back to the end of the line and try again. You only get in once you show a valid one.

  • Ask the user for input
  • Check if the input meets your requirements
  • If it does not, tell them what is wrong and ask again
  • Only proceed once the input is acceptable

This is one of the most practical uses of a while loop. You cannot know in advance how many bad attempts the user will make, so a for loop would not work here.

When Code Goes Wrong

Every programmer makes mistakes. Python helps by giving you error messages. They look intimidating, but they follow a consistent format. Read from the bottom up.

Three categories: syntax errors (Python cannot parse the instruction), runtime errors (code made sense but failed during execution), and logic errors (no crash, but wrong answer).

11
Section 3 · Lecture

try/except: The Safety Net

Some errors are predictable. A user might type "abc" when you need a number, or divide by zero. Instead of letting the program crash, you can catch the error and handle it gracefully.

  • Think of a trapeze artist with a safety net. They still perform, but if they fall, they land safely instead of hitting the ground.
  • try wraps the code that might fail
  • except says what to do if it fails
  • If nothing goes wrong, the except block is skipped entirely

This is especially powerful combined with while loops: catch the error, tell the user, and loop back to ask again.

Section 3 · Lecture

Debugging as a Process

Most beginners debug by randomly changing things until the error goes away. That rarely works. Instead, follow a systematic process, like a doctor diagnosing a patient.

  • Step 1: Reproduce. Can you make the bug happen again? What input triggers it?
  • Step 2: Read. Read the traceback bottom-up. What error type? What line?
  • Step 3: Hypothesize. Based on the error and the line, what do you think went wrong?
  • Step 4: Test. Add a print() before that line to check variable values. Is the variable what you expected?
  • Step 5: Fix and verify. Make one small change, then run the code again to confirm.

The key habit: compare what you expect to what actually happens. The gap between those two is the bug.

13

Summary

  • while condition: repeats as long as the condition is True
  • Use for when you have a sequence or known count. Use while when you are waiting for a condition to change.
  • Always make sure something inside the loop moves toward ending it. Otherwise: infinite loop.
  • break exits the loop immediately. continue skips to the next iteration.
  • Input validation: use a while loop to keep asking until you get valid input
  • Read tracebacks bottom up: error type first, then file/line, then the code
  • try/except catches predictable errors instead of crashing
  • Debug logic errors systematically: reproduce, read, hypothesize, test with print statements, fix
