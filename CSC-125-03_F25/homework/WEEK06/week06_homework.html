What to Submit

Section 1: Overview 15 pts

Write 1 paragraph (5-6 sentences): What problem does this solve? Who would use it? What are the main features?

Section 2: Requirements 20 pts

Inputs: What user enters (be specific)
Calculations: What the program computes
Outputs: What the program displays

Section 3: Pseudocode 40 pts

Step-by-step instructions in plain English. Each step should be clear enough that someone could follow it. Must include ALL features from your chosen problem.

Graded on: Completeness (15), Clarity (15), Logical order (10)

Section 4: Test Cases 20 pts

Create 3 test scenarios with sample inputs and expected outputs. Each should test different conditions in your program.

Section 5: Reflection 5 pts

Write 1 paragraph: What was challenging? Why did you structure it this way? What would you add if you expanded it?

Pseudocode Guidelines

Good pseudocode is like a recipe: specific steps anyone could follow, written in plain English.

Use action verbs: "Ask user for...", "Calculate...", "Store...", "Display...", "Compare..."

Be specific: "Add all 5 scores and divide by 5" not just "calculate average"

Use clear conditions: "If total is greater than budget" not just "check budget"

Avoid being too vague: Not "Get the data" but rather "Ask user to enter price for Store 1 and store it in price1"

Avoid being too detailed: Not "Initialize float variable x = 0.0" but rather "Create variable to store total"

Additional Tips

  • Write pseudocode like you're explaining to a friend (no Python syntax)
  • Make test cases that try different scenarios (normal, boundary, extreme)
  • Check that your pseudocode covers every requirement from your chosen problem

Coffee Shop Order Calculator

Section 1: Overview

This program helps coffee shop customers quickly calculate their order total including tax. The user enters 3 drink prices, and the program calculates the subtotal, adds 7% tax, displays the final total, and identifies the most expensive drink. This is useful for customers who want to know their total before ordering or for tracking spending. It helps people make informed decisions about their purchases and stay within budget.

Section 2: Requirements

Inputs: Price of drink 1 (number), Price of drink 2 (number), Price of drink 3 (number)

Calculations: Subtotal (sum of all prices), Tax amount (subtotal × 0.07), Final total (subtotal + tax), Maximum price (highest of the 3 prices)

Outputs: Subtotal, Tax amount, Final total, Most expensive drink price

Section 3: Pseudocode

Display "Welcome to Coffee Shop Order Calculator"

Create an empty list called prices

Ask user "Enter price of drink 1:"
Read price from user and add it to the prices list

Ask user "Enter price of drink 2:"
Read price from user and add it to the prices list

Ask user "Enter price of drink 3:"
Read price from user and add it to the prices list

Calculate subtotal by finding the sum of all values in the prices list
Calculate tax as subtotal times 0.07
Calculate total as subtotal plus tax

Display "Subtotal: $" followed by subtotal
Display "Tax (7%): $" followed by tax
Display "Total: $" followed by total

Find the maximum value in the prices list
Store this maximum value in max_price

Display "Most expensive drink: $" followed by max_price

If max_price is greater than 6:
    Display "Consider a smaller size to save money"

Display "Thank you for your order!"

Section 4: Test Cases

Test 1 - Normal order:
Inputs: $4.50, $3.25, $5.00
Expected: Subtotal $12.75, Tax $0.89, Total $13.64, Most expensive $5.00, No suggestion shown

Test 2 - Expensive drink:
Inputs: $3.00, $7.50, $4.00
Expected: Subtotal $14.50, Tax $1.02, Total $15.52, Most expensive $7.50, "Consider a smaller size" message shown

Test 3 - All same price:
Inputs: $4.00, $4.00, $4.00
Expected: Subtotal $12.00, Tax $0.84, Total $12.84, Most expensive $4.00, No suggestion

Section 5: Reflection

The most challenging part was deciding how to handle finding the maximum price among three values, since I had to think through the comparison logic carefully. I structured the pseudocode to collect all inputs first, then do all calculations, then display all outputs. This mirrors how someone would actually order (tell the cashier everything, then they calculate and tell you the total). An alternative approach would be to calculate and display after each input, but that seemed less natural. If I expanded this program, I would add a feature to track multiple orders throughout the day and calculate daily spending totals.
