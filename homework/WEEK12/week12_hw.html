Your program must solve a real problem, provide useful information, or create an engaging interaction. Think about:

Required Technical Components

Your program MUST include all of these:

  • Dictionary: Store and retrieve data using at least one dictionary (with at least 3 key-value pairs)
  • Function: At least one custom function you create (must have parameters and/or return value)
  • Loop: Use for or while loop for repetition (processing data, menu system, or validation)
  • Conditional Logic: if/elif/else statements for decision making
  • User Input: Accept input from the user (with validation when appropriate)
  • Meaningful Output: Display results, calculations, or information clearly

Scope Guidelines: Your program should be substantial enough to demonstrate your skills but not so complex that you cannot complete it in 2-3 weeks. Aim for 75-150 lines of code when finished. Programs that are too simple (under 50 lines) or overly ambitious (over 200 lines) will receive feedback to adjust scope.

Thinking About Dictionaries

Since dictionaries are required, plan how you will use them effectively:

Good dictionary uses:

  • Menu items with prices: {"burger": 12.99, "fries": 4.99, "soda": 2.99}
  • Student records: {"Alice": {"gpa": 3.8, "major": "CS"}, "Bob": {...}}
  • Quiz questions and answers: {"What is 2+2?": "4", "Capital of France?": "Paris"}
  • Product inventory: {"laptop": {"price": 999, "quantity": 5}, ...}
  • Translation tool: {"hello": "hola", "goodbye": "adiós", ...}

Thinking About Functions

Functions organize your code and make it reusable. Consider what tasks can be separated into functions:

Good function examples:

  • Calculation function: calculate_total(price, quantity, tax_rate)
  • Validation function: is_valid_choice(choice, options)
  • Processing function: find_highest_grade(grades_dict)
  • Display function: print_menu(items_dict)
  • Conversion function: celsius_to_fahrenheit(temp)

Pseudocode Guidelines

Good pseudocode is specific but not code: Write steps someone could follow, using plain English, not Python syntax.

Use action verbs: Ask, Store, Calculate, Display, Check, Compare, Loop through, Call function

Be specific about data: Not "get input" but "Ask user for item name and store in variable"

Show dictionary operations clearly: "Create dictionary with menu items as keys and prices as values"

Show function calls: "Call calculate_total function with price and quantity, store returned value"

Avoid being too vague: Not "process the data" but "Loop through each key in dictionary and calculate sum of values"

Avoid Python syntax: Not "for item in menu:" but "For each item in the menu dictionary"

Pseudocode Example

Bad pseudocode (too vague):

Get menu
Process order
Calculate total
Show result

Good pseudocode (specific and clear):

Create dictionary called menu with items as keys and prices as values:
    "burger" maps to 12.99
    "fries" maps to 4.99
    "soda" maps to 2.99

Display "Welcome to Restaurant Order System"

Create empty list called order_items

Ask user "How many items do you want to order?" and store in num_items

Loop num_items times:
    Display all menu items with their prices
    Ask user to enter an item name
    Check if item exists in menu dictionary:
        If yes: add item to order_items list
        If no: display "Item not found" and ask again
    
Define function calculate_total that takes order_items list and menu dictionary:
    Set subtotal to 0
    For each item in order_items:
        Look up price in menu dictionary
        Add price to subtotal
    Calculate tax as subtotal times 0.08
    Calculate final_total as subtotal plus tax
    Return final_total

Call calculate_total function with order_items and menu
Store returned value in total

Display "Order Summary:"
Display each item in order_items
Display subtotal
Display tax amount
Display final total

Tips for Success

  • Start by deciding what your dictionary will store (this often drives the whole program)
  • Plan your function early (what calculation or task can be separated out?)
  • Write pseudocode that YOU can understand and follow when you start coding
  • Test cases should cover normal use, edge cases, and potential errors
  • Simple and complete is better than complex and unfinished
  • If you cannot explain how something works, you need to revise it
